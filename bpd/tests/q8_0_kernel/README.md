# q8_0 dot kernel - Prolog->LLVM (no Python in the toolchain)

The q8_0 x q8_0 block-dot kernel GENERATED by Prolog (lib/prolog_to_llvm.pl emit_q8_0_dot/2),
compiled by llc, executed, measured vs the ggml/llama.cpp dump. Dependency surface: swipl + llc.

## Pipeline (no Python)
  swipl -g "emit_q8_0_dot(F, avx), halt" lib/prolog_to_llvm.pl   # Prolog emits LLVM IR
  llc -mattr=+avx,+f16c,-avx2,-fma -filetype=obj q8dot.ll -o q8dot.o   # llc compiles
  gcc -mf16c q8dot_harness.c q8dot.o -lm -o test                      # link (harness = throwaway)

## Status (measured vs ggml dump = the Ollama target)
- q8dot_micro.c (single block, traced): kernel = 0 ULP vs hand-computed reference (arithmetic PROVEN).
- q8dot_harness.c (full o_proj): 1.49e-8 vs ggml dump.
- Remaining 1.49e-8 = fp ACCUMULATION GROUPING: my kernel compiles to scalar imull/vmulss/fadd
  (sequential); ggml compiles to vpmaddubsw/vpmaddwd/vcvtdq2ps/vmulps/vaddps (8-lane vector, two-block).
  Integer dot identical (exact); fp scale+accumulate ORDER differs. Matching 8-lane accumulation -> 0 ULP.

## CPU note (critical)
Xeon E5-2697 v2 = IVY BRIDGE: avx + f16c + sse4, NO avx2, NO fma. ggml compiles the AVX path
(mul_ps+add_ps, NOT fmadd; two-block pairing). Compiling -mavx2/-mfma SIGILLs. q8_0 quant:
id = 1.0f/d with d FP32 (amax/127), y.d STORED as fp16(d).

## BIT-EXACT VERIFICATION (2026-05-30) — 0 ULP vs ggml dump

singlerow_verify.py reproduces ggml's q8_0 mul_mat output bit-for-bit (0 ULP)
for a single output element: weight row0 of blk.0.attn_q · activation token0.

### The decisive finding: quantizer rounding
The intrinsic kernel (bpd_q8_0_dot_intrinsic.ll) was ALREADY bit-exact — int dot,
AVX lane layout, mul-then-add accumulation, and hsum_float_8 all match ggml's
__AVX__ path. The residual (was 85 ULP / 1.27e-6) lived entirely in the
ACTIVATION QUANTIZER:

- ggml's SCALAR quantize_row_q8_0_ref uses roundf() = round half AWAY from zero.
- ggml's DISPATCHED SIMD quantize_row_q8_0 (the one that ACTUALLY RUNS) uses
  _mm256_round_ps(_MM_FROUND_TO_NEAREST_INT) = round half to EVEN.

To match ggml bit-exactly, the activation quantizer MUST mirror the SIMD path:
1. amax = running max of fabsf(x) in fp32
2. d = amax / 127.0f   (fp32)
3. id = 1.0f / d       (fp32 — f64 idv causes double-rounding errors)
4. x0 = x * id         (fp32 — the f32 multiply can land EXACTLY on a .5 tie
   that f64 misses; the decisive lane was x*id = 22.5 exactly -> even -> 22)
5. round half to EVEN (np.rint / _mm256_round_ps NEAREST_INT), not roundf
6. store scale as GGML_FP32_TO_FP16(d)

This is a GENERAL lesson: to match ggml quantized matmul bit-exactly, replicate the
SIMD-dispatched quantizer (fp32 ops + round-half-even), NOT the scalar _ref.

### Diagnostic tools
- ggml_dot_check.c — links real libggml-cpu.so, calls ggml_vec_dot_q8_0_q8_0
  on supplied inputs (needs ggml_cpu_init() or returns 0). Used to dump ggml's
  exact quants and prove the single-row dot reproduces the dump (cross-row was a
  red herring).
- q8quant_check.c — replicates the quantizer in C with F16C (_cvtss_sh).
